# Implementation Plan - Maya Production Pipeliner

> Before validation, production needs readable structure.

## Purpose

This document defines the build sequence for Maya Production Pipeliner.

The project is implemented in small vertical slices. Each phase should add one clear layer of functionality, keep the repository import-safe, and avoid claiming runtime behavior before it is backed by code and manual test evidence.

## Current implementation status

The repository is currently in scaffold / implementation-in-progress state.

Phase 1 created the package structure, placeholder modules, README, license, examples, manual test checklist, and install helper.

A Phase 1 correction was identified for import safety: Maya runtime imports should not break package imports outside a live Maya session, and importing the package should not open UI or mutate a scene.

## Build principles

* Build in small phases.
* Do not implement the full tool in one pass.
* Keep code import-safe where practical.
* Keep Maya scene mutation restricted to Apply mode.
* Keep Dry Run read-only.
* Keep UI feedback based on `RunResult`.
* Keep full route detail in TXT/JSON reports.
* Do not expand product scope during implementation.
* Do not mark a phase complete until it has matching evidence.

## Phase 1 Correction - Import Safety

Goal: prevent scaffold imports from requiring a live Maya runtime.

Acceptance criteria:

* importing `maya_production_pipeliner` does not scan or mutate the scene;
* importing package modules does not open UI;
* Maya imports are guarded or deferred;
* missing Maya runtime does not break non-runtime inspection where practical;
* runtime functions that require Maya fail clearly or remain unavailable outside Maya.

---

## Phase 2 - Data Contracts and Configuration

Goal: define constants and core data structures.

Expected work:

* define scope modes;
* define execution modes;
* define group names;
* define operation status values;
* define report defaults;
* define `max_ui_preview_items`;
* define JSON-safe `ObjectRecord`, `RouteDecision`, and `RunResult` structures;
* centralize names in `config.py`.

Acceptance criteria:

* data contracts are JSON-safe;
* no live Maya objects are stored in records;
* route and status values match the frozen scope;
* tests or simple examples can instantiate the records.

---

## Phase 3 - Reporter

Goal: implement TXT and JSON report writing without scene mutation.

Expected work:

* create report path resolver;
* support scene directory, workspace, and temp fallback;
* write TXT report;
* write JSON report;
* include warnings, routes, operation status, report-only state, and summary data;
* update example reports when generated by real code.

Acceptance criteria:

* reporter can write valid TXT/JSON from provided data;
* report paths are returned to caller;
* reporter does not inspect UI state;
* reporter does not mutate the Maya scene.

---

## Phase 4 - Scanner

Goal: collect scene facts into `ObjectRecord`.

Expected work:

* scan All Scene scope;
* scan Selected scope;
* scan Visible scope;
* normalize selected shapes, components, or child nodes when practical;
* collect transform and shape data;
* detect material state;
* detect default material;
* detect multi-material state;
* detect references;
* detect instances;
* detect rig/deformer indicators;
* detect previous tool output;
* resolve visibility where practical.

Acceptance criteria:

* scanner collects facts only;
* scanner does not classify;
* scanner does not move or create scene objects;
* Visible scope avoids obvious false positives where practical.

---

## Phase 5 - Classifier

Goal: create safety-aware route decisions.

Expected work:

* apply classification priority;
* set route and target group;
* set `can_move`;
* set `report_only`;
* set `preserve_reason`;
* set initial `operation_status`;
* route material review cases;
* route unclear cases;
* preserve references, instances, rig/deformer-sensitive content, and unsafe unclear content.

Acceptance criteria:

* classifier does not mutate the scene;
* route decisions contain enough information for Dry Run, Apply, reports, and tests;
* unsafe content is preserved by default.

---

## Phase 6 - Dry Run Pipeline

Goal: connect scanner, classifier, reporter, and `RunResult` without modifying the scene.

Expected work:

* implement pipeline settings;
* orchestrate scan and classify;
* generate route plan;
* write reports;
* return lightweight `RunResult`;
* ensure Dry Run creates no groups and moves no objects.

Acceptance criteria:

* Dry Run is read-only;
* `RunResult` includes summary, warnings, report paths, route count, and preview routes;
* reports contain full route details;
* UI is not required for pipeline execution.

---

## Phase 7 - Minimal UI and Launcher

Goal: provide a simple Maya-native interface.

Expected work:

* create UI controls for scope mode;
* create UI controls for execution mode;
* create editable ignore string field;
* add run button;
* call pipeline;
* display lightweight `RunResult` summary;
* display warnings and report paths.

Acceptance criteria:

* UI does not parse TXT/JSON reports for state;
* UI does not render complete route lists in heavy scenes;
* launcher imports safely and opens UI only when `launch()` is called.

---

## Phase 8 - Safe Apply Organizer

Goal: execute route plan safely.

Expected work:

* create or reuse `Pipeline_Organized`;
* create or reuse child groups;
* move only decisions with `can_move = true`;
* preserve report-only content;
* detect `already_in_target`;
* validate node existence before movement;
* update `new_long_name`;
* set `did_move`;
* set final `operation_status`;
* record failures without corrupting the scene.

Acceptance criteria:

* Apply follows the route plan;
* Apply does not move references, instances, rig/deformer-sensitive content, or unsafe unclear cases;
* repeated runs do not duplicate structure;
* long-name mutation is tracked.

---

## Phase 9 - Hardening and Release Polish

Goal: prepare the public repository for a functional milestone.

Expected work:

* run manual test checklist;
* update README status;
* update examples with real generated reports;
* add screenshots or GIFs only if they reflect actual behavior;
* verify import safety;
* verify Dry Run;
* verify Apply on safe scenes;
* verify preservation behavior;
* verify report accuracy;
* verify documentation links.

Acceptance criteria:

* public documentation matches implementation state;
* no functional claim is made without evidence;
* first usable milestone is clearly identified;
* release status is honest.

## First Functional Milestone

The first functional milestone is the Dry Run vertical slice:

```text
scan a simple scene
-> classify route decisions
-> return lightweight RunResult
-> write TXT/JSON reports
-> make no scene modifications
```

This milestone is the first point where the project can demonstrate behavior without risking scene mutation.

## Commit Strategy

Recommended commit sequence:

```text
chore: seed repository scaffold
fix: make scaffold imports safe outside Maya
docs: add public planning and architecture docs
feat: add data contracts and config constants
feat: add report writer
feat: add scene scanner
feat: add safety-aware classifier
feat: add dry-run pipeline
feat: add Maya UI and launcher
feat: add safe apply organizer
docs: update examples and manual test results
```

Commit messages should describe actual changes. Avoid marking unfinished functionality as complete.

Before presenting a release candidate:

* package imports cleanly;
* Dry Run works on simple scenes;
* reports are generated from actual run data;
* Apply works on safe simple scenes;
* references and sensitive content remain preserved;
* repeated execution is idempotent;
* UI remains lightweight;
* documentation matches implementation state.
